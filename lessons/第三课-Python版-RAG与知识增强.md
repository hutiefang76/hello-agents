## 第一章 背景问题：当LLM不知道你的私有文档时 ⭐⭐⭐⭐⭐

### 1.1 三类LLM解决不了的场景

GPT-4再厉害，也有它根本回答不了的问题。这不是能力问题，而是**知识边界**问题。理解这个边界，是理解RAG存在价值的第一步。

LLM的知识来源于训练数据，而训练数据有三个天然缺陷：**时效性截止、私有性缺失、精确性不足**。对应地，有三类场景是LLM天然解决不了的。

---

**场景一：实时信息——今日新闻、实时股价、最新政策**

```
用户：今天A股市场涨了多少？
LLM：我的训练数据截止到2024年初，无法提供今日实时数据。
```

LLM就像一本"印刷完成的百科全书"——印刷之后发生的事情，它一无所知。

具体业务场景：
- 金融客服：用户问"今日人民币汇率是多少？"
- 新闻摘要助手：用户问"今天有什么重大新闻？"
- 政策咨询机器人：用户问"最新的个税起征点是多少？"
- 法律助手：用户问"最高法最新的司法解释是什么？"

这类场景的共同特点：**知识更新频率极高，远超模型重训练的频率**。任何依赖LLM内置知识的方案都会失效。

---

**场景二：私有知识——企业合同、内部代码库、医院病历、法院判决**

```
用户：我们公司和XX供应商的合同里，交货违约金条款是怎么写的？
LLM：抱歉，我没有访问贵公司内部合同的权限，无法回答。
```

这类信息从未出现在任何公开互联网上，LLM的训练集里根本不可能包含它。

具体业务场景：
- **企业智能客服**：FAQ知识库、产品手册、合同条款，都是私有的
- **代码库助手**：公司内部的私有代码库、架构文档、API规范
- **医疗辅助诊断**：患者病历、医院内部诊疗规范、科室操作手册
- **法律检索助手**：特定法院的历史判决、律所内部案例库
- **政府服务机器人**：地方政府政策文件、办事流程、内部规定

这类场景的共同特点：**数据永远不会出现在公开训练集里，且数据量通常在百GB到TB级别**，远超任何Prompt窗口。

---

**场景三：精确事实——需要引用具体条款、精确数字、来源可溯**

```
用户：根据《劳动合同法》第四十七条，经济补偿金怎么计算？
LLM（可能的回答）：经济补偿金按工龄计算，每满一年支付一个月工资...
```

这个回答看起来对，但有两个致命问题：
1. LLM可能混淆了不同修订版本的条款
2. 没有来源引用，无法核实——这在法律、医疗、金融场景是不可接受的

LLM的"幻觉"（Hallucination）在精确事实查询上表现得尤为危险。当用户需要的是"原文第X条写的是什么"，而不是"大概意思是什么"时，LLM的语言模型本质就成了障碍——它倾向于生成"听起来对"的内容，而不是"原文是这样"的精确引用。

精确事实场景的特点：
- 需要精确引用原文，而非语言模型的"理解和转述"
- 答案必须可溯源、可验证
- 错误的代价高（医疗误诊、法律错引）

---

**小结：三类场景的共同特点**

```
┌─────────────────────────────────────────────────────────────┐
│              LLM的三大知识边界                               │
├───────────────┬──────────────────────────────────────────────┤
│ 时效性        │ 训练截止日期后的信息，LLM无法知晓            │
│ 私有性        │ 未出现在公开训练集的信息，LLM无法知晓        │
│ 精确性        │ 需要原文引用和来源溯源，LLM容易产生幻觉      │
└───────────────┴──────────────────────────────────────────────┘
```

这三类问题有一个共同的解法：**在LLM生成答案之前，先从外部知识库中检索相关信息，然后把检索到的内容注入Prompt**。这就是RAG的核心思想。

---

### 1.2 从暴力方案到RAG的演进链

理解一个技术，最好的方式是理解"它之前的方案为什么不行"。RAG不是凭空出现的，它是在经历了几次失败尝试之后的工程演进结果。

---

**方案一：暴力Prompt注入——把所有文档塞入Prompt**

最直觉的想法：把所有文档都塞进Prompt，让LLM直接在里面找答案。

```
┌─────────────────────────────────────────────────────────────┐
│ 方案一：暴力Prompt注入                                       │
│                                                              │
│  [System: 你是XX公司的智能助理]                             │
│  [文档全文：100万字的产品手册和合同文本...]                  │
│  [用户问题：...]                                             │
│                                                              │
│  问题一：100万字塞不进Context Window（即使GPT-4也只有128K） │
│  问题二：即使塞进去了，推理成本极高（按Token计费）           │
│  问题三：LLM在超长上下文中的注意力会发散，答案质量下降      │
│  问题四：文档更新时，每次都要重新组织整个Prompt              │
└─────────────────────────────────────────────────────────────┘
```

实际上，即使Context Window足够大（如Claude的200K），这个方案也存在"Lost in the Middle"问题——研究表明，LLM对Context中间部分的注意力显著弱于开头和结尾。把100万字的文档全塞进去，LLM往往"看了但没看进去"。

**结论**：成本不可接受，且质量随文档长度下降。

---

**方案二：Fine-tuning——把知识"烧"进参数**

既然Prompt放不下，那就把知识训练到模型参数里，让模型"记住"这些知识。

```
┌─────────────────────────────────────────────────────────────┐
│ 方案二：Fine-tuning（微调）                                  │
│                                                              │
│  原始LLM ──── [私有知识数据集训练] ──── 专用LLM             │
│                                                              │
│  听起来很美，但有三个致命问题：                              │
│                                                              │
│  问题一：成本高                                              │
│    - 即使是LoRA微调，也需要GPU资源和数据工程                 │
│    - 对小公司而言，成本不可持续                              │
│                                                              │
│  问题二：知识更新需要重新训练                                │
│    - 合同修订了 → 重训练                                     │
│    - 政策更新了 → 重训练                                     │
│    - 这意味着知识库不能实时更新                              │
│                                                              │
│  问题三：知识边界不清，幻觉依然存在                          │
│    - Fine-tune的LLM仍然会"自信地编造"                       │
│    - 无法区分"训练数据里有的"和"自己推理出来的"             │
│    - 无法提供来源引用                                        │
└─────────────────────────────────────────────────────────────┘
```

Fine-tuning更适合**改变模型的行为和风格**（如让模型更简洁、使用特定格式），而不是**注入特定知识**。这个认知很重要，第六章会详细展开。

**结论**：Fine-tuning是改变"能力"的工具，不是注入"知识"的工具。

---

**方案三：RAG——检索增强生成**

RAG的核心洞察：**不需要把所有知识都准备好，只需要在用户提问时，实时检索出最相关的那部分知识，然后注入Prompt。**

```
┌─────────────────────────────────────────────────────────────┐
│ 方案三：RAG（检索增强生成）                                  │
│                                                              │
│  ┌──────────────────────────────────────────────────┐       │
│  │              外部知识库（向量数据库）              │       │
│  │   [合同Chunk1] [合同Chunk2] ... [手册Chunk100]   │       │
│  └────────────────────┬─────────────────────────────┘       │
│                        │ 实时检索Top-K相关片段               │
│  用户问题 ────────────▶│                                     │
│                        ▼                                     │
│          [相关Chunk1] + [相关Chunk2] + [用户问题]           │
│                        │                                     │
│                        ▼                                     │
│                       LLM生成答案（含来源引用）              │
└─────────────────────────────────────────────────────────────┘
│                                                              │
│  解决了什么：                                                │
│  ✓ 成本：只检索Top-K个片段，Prompt保持精简                  │
│  ✓ 时效性：更新知识库即可，不需要重训练模型                  │
│  ✓ 私有性：知识库放在自己的服务器上，完全可控                │
│  ✓ 精确性：答案有来源引用，幻觉风险大幅降低                  │
└─────────────────────────────────────────────────────────────┘
```

**RAG的演进链总结**：

```
暴力Prompt注入
      │
      │ 问题：Context Window限制，成本不可接受
      ▼
Fine-tuning知识注入
      │
      │ 问题：训练成本高，无法实时更新，无来源引用
      ▼
RAG（检索增强生成）
      │
      │ 优势：按需检索，实时更新，成本可控，有来源引用
      ▼
    工程实践主流方案
```

RAG的本质是一种**"先搜索，再生成"**的架构模式。它把知识管理的问题从"模型训练阶段"转移到了"推理阶段"，这个转移带来了极大的工程灵活性。

### 📝 第一章面试题

---

**面试题1：什么是RAG？它解决了LLM的哪些核心问题？**

> 💡 **面试金句**：RAG是"检索增强生成"，本质上是给LLM配了一个实时的外部知识库。它解决了LLM三个核心局限：知识截止日期（时效性）、私有数据不可见（私有性）、以及无来源引用导致的幻觉风险（精确性）。技术上，RAG先把私有文档向量化存入数据库，用户提问时实时检索Top-K相关片段，将片段拼入Prompt后交给LLM生成答案。

---

**面试题2：为什么不直接把所有文档放到Prompt里，而要用RAG？**

> 💡 **面试金句**：这是一个看起来简单但有深度的问题。直接塞文档有三个问题：第一，Context Window有上限，百万字文档物理上塞不进去；第二，即使塞进去，推理成本按Token计费，代价极高；第三，存在"Lost in the Middle"问题——LLM对超长上下文中间部分的注意力会显著下降，质量不可控。RAG通过语义检索，只把最相关的Top-K片段注入Prompt，在精简输入的同时保证了相关性，是成本和质量的最优平衡。

## 第二章 Embedding向量：让机器理解"相似性" ⭐⭐⭐⭐⭐

### 2.1 什么是Embedding？

RAG的核心能力是**语义检索**——不是找"包含相同关键词"的文档，而是找"语义上相关"的文档。这种能力来自于Embedding（向量嵌入）技术。

---

**直觉类比：把文本映射到"语义空间"的坐标**

想象一个巨大的多维空间，每个词、每个句子都是这个空间里的一个点（坐标）。语义相似的词/句子，在这个空间里距离很近；语义差异大的词/句子，距离很远。

```
        ┌──────────────────────────────────────────────────────┐
        │                  语义空间（简化为2D示意）             │
        │                                                      │
        │  动物区域：                                          │
        │    猫 ●──● 喵星人                                   │
        │    ●                                                 │
        │    狗 ●──● 汪星人                                   │
        │         \                                            │
        │          ● 宠物                                      │
        │                                                      │
        │                      交通工具区域：                  │
        │                         ● 汽车──● 轿车              │
        │                         ● 卡车                       │
        │                                                      │
        │  （猫和汽车之间的距离远大于猫和狗之间的距离）        │
        └──────────────────────────────────────────────────────┘
```

Embedding模型做的事情就是：**把任意文本，转换成这个语义空间里的一个坐标（向量）**。

---

**维度的直觉理解**

OpenAI的`text-embedding-ada-002`输出1536维向量。1536维不是1536个坐标轴，更好的理解方式是：**1536个语义特征**。

每个维度可以近似理解为捕捉了文本的某一个语义方面：
- 某些维度捕捉"情感极性"（正面/负面）
- 某些维度捕捉"主题领域"（技术/法律/医疗）
- 某些维度捕捉"文体风格"（正式/口语）
- ...以此类推（但实际上这些维度没有人类可解释的标签）

维度越高，语义表达能力越强，但计算和存储成本也越高。这就是为什么有的模型提供"可压缩"的Embedding——在维度和精度之间做权衡。

---

**为什么Embedding能理解语义？**

Embedding来自大规模语言模型的预训练过程。模型在海量文本上训练时，会学到"哪些词经常出现在相似的上下文中"（共现关系）。

经典例子：
- "猫"和"喵星人"经常出现在相同的上下文（描述宠物的文章），所以它们的向量相近
- "国王"减去"男性"加上"女性"，结果接近"女王"——这是向量运算反映语义关系的著名例子

预训练阶段积累的共现关系，使得Embedding向量天然具备语义聚合的能力。这是Embedding在RAG中发挥作用的根本原因。

---

### 2.2 相似度计算方法对比（余弦/点积/欧氏）

有了Embedding向量，下一步是：如何衡量两个向量的"相似程度"？有三种主流方法。

| 方法 | 计算方式 | 直觉 | 适用场景 | 注意事项 |
|------|----------|------|----------|----------|
| 余弦相似度 | 两个向量夹角的余弦值，范围[-1,1] | 只看"方向"，不看"长度" | RAG检索的默认首选，对向量长度不敏感 | 需归一化，计算略慢 |
| 点积相似度 | 两个向量各维度乘积之和 | 同时考虑"方向"和"长度" | 当向量已经归一化时，等价于余弦相似度；适合排序场景 | 向量长度差异大时结果偏差大 |
| 欧氏距离 | 两个向量在空间中的"直线距离" | 直接量距离，越近越相似 | 图像检索、聚类分析 | 对向量长度敏感，RAG中用得较少 |

---

**为什么RAG首选余弦相似度？**

余弦相似度的核心优势是**对向量长度不敏感**。一篇100字的短文和一篇5000字的长文，即使内容相似，它们的向量长度（模）可能差异很大。用欧氏距离会因为长度差异产生偏差，而余弦相似度只关注方向，不受长度影响。

```
余弦相似度 = (A · B) / (|A| × |B|)

当两个向量方向完全相同时，余弦相似度 = 1（最相似）
当两个向量方向垂直时，余弦相似度 = 0（无关）
当两个向量方向相反时，余弦相似度 = -1（最不相似）
```

在实际RAG系统中，Embedding模型通常会输出归一化的向量（模为1），此时点积和余弦相似度等价，可以用点积替代以获得更快的计算速度。

### 2.3 主流Embedding模型选型（含中文优化）

选择Embedding模型时，需要考虑：维度、中文支持、成本、是否可私有部署。

| 模型 | 维度 | 特点 | 中文支持 | 适用场景 |
|------|------|------|----------|----------|
| OpenAI text-embedding-ada-002 | 1536 | 综合性能好，API简单，最常用 | 一般 | 海外产品，快速原型，英文为主 |
| OpenAI text-embedding-3-small | 1536（可压缩至512） | 更便宜，支持维度压缩（Matryoshka） | 一般 | 成本敏感场景，大规模检索 |
| OpenAI text-embedding-3-large | 3072 | 最高精度，最贵 | 较好 | 精度要求极高的场景 |
| Alibaba text-embedding-v3 | 1024 | 阿里云API，中文优化，多语言 | 优秀 | 国内产品，中英混合文档 |
| BGE-large-zh（开源） | 1024 | 完全开源，中文SOTA，可私有部署 | 最佳 | 私有化部署，中文场景，成本敏感 |

---

**选型决策指南**

```
中文场景或数据安全要求高？
      │是                         │否
      ▼                            ▼
私有化部署要求？             使用OpenAI Embedding
      │是         │否             （ada-002或3-small）
      ▼           ▼
BGE-large-zh   Alibaba
（开源免费）   text-embedding-v3
```

**BGE（BAAI General Embedding）为何值得重视**：
- 北京人工智能研究院（BAAI）开源
- 在中文Embedding评测榜单（MTEB中文）长期排名前列
- 支持指令式Embedding（通过前缀指令区分检索和文档两种任务）
- 完全开源，可在企业内部私有部署，数据不出境

---

**Embedding模型的通用选型原则**

1. **先考虑数据安全**：金融、医疗、法律场景数据不能上云，必须选开源模型私有部署（BGE系列）
2. **再考虑中文效果**：纯英文或多语言场景，OpenAI系列足够；中文为主，选中文优化模型
3. **最后考虑成本**：高频检索场景考虑向量维度压缩（3-small的Matryoshka特性）

### 📝 第二章面试题

---

**面试题1：什么是文本Embedding？为什么它能表示语义相似性？**

> 💡 **面试金句**：Embedding是把文本映射到高维语义空间的坐标向量。它能表示语义相似性，根本原因在于预训练时学到的"共现关系"——在海量文本中，语义相似的词和句子会出现在相似的上下文环境中，模型通过自监督学习将这种共现关系编码到向量空间，使得语义相近的文本向量方向相近，可以用余弦相似度等度量来量化。

---

**面试题2：余弦相似度和欧氏距离在RAG中各有什么优劣？**

> 💡 **面试金句**：余弦相似度只关注向量方向（夹角），对向量长度不敏感，因此短文和长文只要语义相近就能被匹配到，这在RAG中非常重要——用户的短问题和文档的长片段在长度上差异很大，但语义可能很相关。欧氏距离同时受方向和长度影响，适合图像检索等场景。RAG默认选余弦相似度，当向量归一化后，点积和余弦等价，点积计算更快，可以替代。

---

**面试题3：为什么中文RAG需要选择中文优化的Embedding模型？**

> 💡 **面试金句**：通用Embedding模型（如OpenAI ada-002）在英文语料上训练更多，中文语义的捕捉能力相对较弱。中文有独特的字词粒度问题（字vs词vs短语）、以及大量需要上下文理解的语义模糊性。BGE-large-zh等中文优化模型在中文MTEB评测上显著超过通用模型，特别是在中文问答、检索类任务上。对于企业级中文RAG，选对Embedding模型往往比优化分块策略更有效。

---

## 第三章 向量数据库：语义检索的基础设施 ⭐⭐⭐⭐

### 3.1 为什么普通数据库不够用？

有了Embedding向量，下一步是存储它们并支持高效检索。为什么不直接用MySQL或Elasticsearch？

---

**三类数据库的能力边界对比**

| 数据库类型 | 检索方式 | 能力边界 | 经典例子 | 语义理解 |
|------------|----------|----------|----------|----------|
| SQL关系型数据库 | 精确匹配（WHERE、LIKE） | 精确字段匹配，支持模糊字符串匹配 | "猫"查不到"喵星人" | 无 |
| 全文检索（Elasticsearch） | 关键词倒排索引（BM25算法） | 关键词匹配，支持同义词词典扩展 | 词典配置后可以，但维护成本高 | 有限 |
| 向量数据库 | 向量相似度（ANN搜索） | 语义相似度匹配，无需词典配置 | 查"猫"自动找到"喵星人"、"宠物猫" | 天然支持 |

---

**一个具体例子**

场景：用户问"员工离职后的竞业协议怎么处理？"

- SQL数据库：只能找到文字中包含"竞业"或"离职"的记录，无法理解"竞业协议"和"竞业限制条款"是同一件事
- Elasticsearch（不配同义词词典）：同上，依赖关键词精确匹配
- 向量数据库：能检索到"竞业限制条款"、"竞业禁止协议"、"非竞争协议"相关的内容，因为这些概念在语义空间里距离很近

向量数据库的核心价值：**把"语义相似"转化为可量化的距离，不需要人工配置同义词表，天然理解语义变体。**

---

**图书馆类比**

```
传统数据库 = 按书号（ISBN）查书
           → 必须知道精确的书号才能找到

全文检索   = 按书名关键词查书
           → "Python机器学习"和"Machine Learning with Python"
             需要同义词配置才能互相匹配

向量数据库 = 按主题语义查书（主题索引）
           → "机器学习"、"深度学习"、"神经网络"
             自动归到相邻区域，语义检索天然支持
```

---

### 3.2 ANN近似最近邻搜索（HNSW/IVF直觉）

向量数据库面临的核心技术挑战：**在百万乃至亿级向量中，快速找到最相似的Top-K个向量**。

---

**为什么不用精确KNN（K近邻）？**

精确KNN搜索：把查询向量与数据库里每一个向量都算一次相似度，取最高的K个。

```
时间复杂度：O(N × D)
N = 向量数量，D = 向量维度

实际代入数字：
  N = 100万，D = 1536维
  每次查询需要计算 1,000,000 × 1,536 = 15.36亿 次浮点运算
  延迟：秒级甚至分钟级
  → 完全不可用于生产
```

解决方案：**ANN（Approximate Nearest Neighbor，近似最近邻）搜索**——用少量精度损失，换取数量级的速度提升。

---

**HNSW（Hierarchical Navigable Small World）——分层可导航小世界图**

HNSW是目前最主流的ANN算法，被Pinecone、Weaviate、Chroma等主流向量数据库采用。

直觉类比：**找人问路——分层缩小范围**

```
精确KNN：挨家挨户问路（问遍所有100万户）
             太慢，不可接受

HNSW：
第1层（粗粒度）：先问"你知道北京哪个区有XX吗？"
      │
      ▼ 定位到：海淀区
第2层（中粒度）：再问"海淀区哪条街？"
      │
      ▼ 定位到：中关村大街
第3层（细粒度）：再问"这条街哪个门牌号？"
      │
      ▼ 定位到：XX大厦5楼
```

HNSW建了多层图结构，每层图都是一个"小世界网络"（任意两点之间通过很少的跳数就能到达）。搜索时从最上层（粗粒度，节点少）开始快速定位大致区域，然后逐层向下精细搜索。

关键参数：
- `ef_construction`：建索引时的搜索范围，越大索引质量越高，建索引越慢
- `ef_search`：查询时的搜索范围，越大召回率越高，查询越慢
- `M`：每个节点的最大连接数，影响图的连通性

**准确率 vs 速度的权衡**：
```
ef_search ↑ → 召回率↑，查询速度↓
ef_search ↓ → 召回率↓，查询速度↑
```

---

**IVF（Inverted File Index，倒排文件索引）——聚类后检索**

IVF的直觉：**图书馆按主题分区**

```
建索引阶段：
  把所有向量做K-Means聚类，分成N个"桶"（Cluster）
  → 类似图书馆把书按"文学"、"科学"、"历史"分区存放

查询阶段：
  先找到最接近查询向量的P个桶
  → 只在这P个桶里做精确搜索
  → 不需要扫描全部向量
```

IVF适合超大规模向量库（亿级以上），Milvus中使用广泛。但需要预先知道聚类数N，且索引建立成本比HNSW高。

---

**HNSW vs IVF 简要对比**

| 对比维度 | HNSW | IVF |
|----------|------|-----|
| 算法思路 | 图遍历，分层缩小范围 | 聚类分桶，只搜相关桶 |
| 查询速度 | 极快 | 快 |
| 内存占用 | 高（图结构） | 低 |
| 适合规模 | 千万级以下 | 亿级以上 |
| 主要使用者 | Pinecone, Weaviate, Chroma | Milvus, FAISS |

---

### 📝 第三章面试题

---

**面试题1：向量数据库和传统关系型数据库的本质区别是什么？**

> 💡 **面试金句**：本质区别在于检索方式。关系型数据库做精确字段匹配，回答的是"等于"问题——找到ISBN等于某值的书。向量数据库做语义相似度匹配，回答的是"最像"问题——找到语义上最相关的Top-K个向量。技术实现上，向量数据库用ANN索引（HNSW或IVF）在高维空间里做近似最近邻搜索，而关系型数据库用B+Tree做有序索引精确查找。两者不是替代关系，向量数据库通常和关系型数据库配合使用——向量库做语义检索，关系库存元数据和业务数据。

---

**面试题2：什么是ANN近似最近邻搜索？为什么不用精确KNN？**

> 💡 **面试金句**：精确KNN需要把查询向量与数据库每个向量都算一次相似度，复杂度是O(N×D)。百万向量级别下延迟秒级以上，不可接受。ANN（近似最近邻）用少量精度损失换取数量级的速度提升。主流算法HNSW构建多层图结构，从粗粒度到细粒度逐层缩小搜索范围，查询复杂度降到O(log N)量级，召回率通常能达到95%以上。实际上95%的召回率已经完全够用——RAG系统的瓶颈往往不在这1-5%的精度损失上。

---

**面试题3：生产环境中你会如何选择向量数据库？**

> 💡 **面试金句**：我的选型框架是"能用现有基础设施就用"。如果已有PostgreSQL，首选PgVector，零新增服务成本，ACID保证，SQL查询灵活。如果已有Elasticsearch，叠加向量检索能力而不是新建向量库。如果是全新项目，数据量千万以下考虑Weaviate（原生混合搜索）或Pinecone（零运维），超过千万考虑Milvus（分布式，国内生态好）。开发阶段用Chroma，五分钟跑通原型，后期迁移成本低。

### 4.2 分块策略（Chunking）详解（5种策略对比）

**分块是RAG中最被低估的环节**。"垃圾进，垃圾出"——分块质量直接决定RAG系统质量的上限，比Embedding模型选型、比向量数据库选型都更关键。

分块的核心矛盾：
- **块太大**：检索精度下降（一个Chunk混了多个主题），且占用更多Token
- **块太小**：上下文不完整（切断了关键的前后文关系），导致LLM无法理解语义

| 分块策略 | 原理 | 优点 | 缺点 | 最适合场景 |
|----------|------|------|------|------------|
| 固定大小分块（Fixed-size） | 按固定字符数（如500字）切分，可设重叠（overlap） | 简单，可控，均匀 | 可能切断句子、段落，语义不连贯 | 均质文档（如新闻正文），快速原型 |
| 按句段分块（Sentence/Paragraph） | 按句号、段落符号切分，保持句子完整 | 语义连贯，不切断句子 | 块大小不均匀，长段落可能过大 | 文学文本、法律条款、需要保留句子完整性的场景 |
| 递归字符分块（Recursive Character） | LangChain默认，依次按"\n\n"→"\n"→". "→" "切分，保证语义层次 | 综合最优，兼顾语义和大小 | 需要调参（chunk_size, overlap） | 通用场景的默认首选，绝大多数文档适用 |
| 语义分块（Semantic Chunking） | 用Embedding计算句子间相似度，相似度骤降处为边界 | 最符合语义边界，高质量 | 计算成本高（每句都要Embedding） | 高质量生产环境，主题跳跃频繁的文档 |
| 父子分块（Parent-Child） | 小块（128字）用于检索，大块（512字）用于上下文注入 | 检索精度高，上下文完整 | 存两套索引，系统复杂度增加 | 高质量RAG，对检索精度和上下文完整性都有要求 |

---

**工程实践中的分块参数经验**

```
Chunk Size（块大小）的经验值：
  - 通用场景：300-500字（中文）/ 512-1024 Token（英文）
  - 问答类：200-300字（短问题，精准匹配）
  - 摘要类：800-1200字（需要完整段落理解）

Overlap（重叠）的经验值：
  - 通常为 chunk_size 的 10%-20%
  - 50字重叠可以有效防止关键信息被切断边界

Metadata的重要性：
  - 每个Chunk必须保存来源信息（文件名、页码、章节）
  - 用于生成答案时的来源引用，也用于过滤检索范围
```

---

### 4.3 检索策略（向量检索/混合检索/HyDE/多查询）

仅有好的分块还不够，检索策略决定了"能不能找到正确的Chunk"。

---

**策略一：朴素向量检索（Naive Retrieval）**

最简单的实现：把用户问题向量化，直接做ANN搜索，返回Top-K。

```
用户问题 → Embedding → ANN搜索 → Top-K Chunks → LLM
```

优点：简单，延迟低
局限：对关键词依赖强的查询（如专有名词、代码片段）效果不好

---

**策略二：混合检索（Hybrid Search）**

结合向量相似度和BM25关键词检索，用RRF（Reciprocal Rank Fusion）融合两路结果。

```
用户问题
    │
    ├── 向量检索 → 语义相关结果 ──────┐
    │                                  ├── RRF融合 → 综合Top-K
    └── BM25关键词检索 → 精确词匹配 ──┘
```

适用场景：
- 包含专有名词、型号编码（如"iPhone 15 Pro Max"）的查询
- 法律条款查询（需要精确匹配条款号）
- 代码检索（变量名、函数名的精确匹配）

这是目前生产环境最主流的检索策略，Weaviate、Elasticsearch的向量混合模式都是基于这个思路。

---

**策略三：HyDE（Hypothetical Document Embeddings，假设文档嵌入）**

HyDE解决一个关键问题：**用户问题和文档的语义空间可能存在偏差**。

```
普通检索的问题：
  用户问题："如何处理员工迟到？"  → 短问句，信息量少
  文档内容："第5.3条规定，员工无故迟到超过30分钟..."  → 长段落，信息量大
  → 两者Embedding的语义偏差较大

HyDE的做法：
  第1步：让LLM根据用户问题生成一个"假设答案"
    → "根据通常的公司规定，员工迟到处理方式如下：
       1. 迟到30分钟以内扣绩效分...
       2. 迟到超过30分钟视为旷工半天..."

  第2步：用这个假设答案的Embedding去检索（而不是原始问题）
    → 假设答案与真实文档的语义分布更接近
    → 检索精度显著提升
```

HyDE的代价：每次检索额外多一次LLM调用（生成假设答案），延迟增加。适合对准确性要求高、延迟容忍度相对宽松的场景。

---

**策略四：多查询检索（Multi-Query Retrieval）**

解决问题：用户问题的表达方式可能只是一种角度，同一个问题有多种表达。

```
原始问题："竞业协议对我有什么限制？"

LLM生成多个查询变体：
  ├── "竞业限制协议的内容有哪些？"
  ├── "员工离职后不能做什么？"
  ├── "非竞争条款的法律约束是什么？"
  └── "竞业禁止协议违约后果？"

对每个变体分别检索 → 合并去重 → 综合Top-K
```

优点：召回率大幅提升，减少单一表达方式带来的漏检
代价：N次LLM调用（生成变体）+ N次向量检索，延迟和成本成倍增加

---

### 4.4 重排序（Reranking）

**粗排（ANN检索）+ 精排（Reranker）是RAG系统质量的双重保障。**

---

**为什么需要重排序？**

ANN向量检索使用Bi-Encoder架构：**先分别把查询和文档向量化，再计算向量相似度**。速度极快，但有一个局限：查询和文档是独立编码的，没有捕捉到"查询和文档之间的交互关系"。

重排序使用Cross-Encoder架构：**把查询和文档拼在一起，让模型同时看两者，计算相关性分数**。精度更高，但速度慢（不能预先计算向量）。

```
Bi-Encoder（检索阶段）：
  Query → [Encoder] → Query向量
  Doc   → [Encoder] → Doc向量
  相似度 = 点积(Query向量, Doc向量)
  → 速度快，可提前建索引，但精度有限

Cross-Encoder（重排序阶段）：
  [Query + Doc] → [Cross-Encoder] → 相关性分数
  → 精度高，但无法预建索引，只能在线计算
```

**类比**：粗排（ANN检索）= 海选（从100万人中选出100人），精排（Reranker）= 面试（从100人中精选出最终的10人）。海选速度快但精度粗，面试速度慢但精度高。两者结合，兼顾效率和质量。

---

**常用重排序模型**

| 模型 | 特点 | 部署方式 |
|------|------|----------|
| Cohere Rerank | 云API，效果优秀，接入简单 | SaaS API |
| BGE-Reranker | 开源，中文效果好，可私有部署 | 自托管 |
| JinaAI Reranker | 多语言，API友好 | SaaS API |

### 📝 第四章面试题

---

**面试题1：请描述RAG的完整流水线，从文档入库到最终回答。**

> 💡 **面试金句**：RAG分离线和在线两阶段。离线阶段：加载文档→清洗→Chunking分块（这步最关键）→Embedding向量化→存入向量数据库（带元数据）。在线阶段：用户问题向量化→ANN检索Top-K相关Chunk→（可选）Reranker精排→注入Prompt（上下文+问题+System指令）→LLM生成答案+来源引用。工程重点在于：离线阶段的分块质量决定系统上限，在线阶段的检索策略决定召回率。

---

**面试题2：分块策略（Chunking）对RAG效果有多大影响？有哪些常用策略？**

> 💡 **面试金句**：分块是RAG中最被低估的环节，"垃圾进，垃圾出"——分块质量直接决定检索质量的上限，任何后续的优化都无法弥补分块阶段的错误。五种主流策略：固定大小（简单，可能切断语义）、按句段（语义连贯但块大小不均）、递归字符分块（LangChain默认，综合最优，生产推荐）、语义分块（质量最高但成本高）、父子分块（小块检索精准+大块上下文完整，高质量场景首选）。核心参数：chunk_size控制块大小，overlap控制边界重叠防止信息截断。

---

**面试题3：什么是混合检索（Hybrid Search）？为什么比纯向量检索效果好？**

> 💡 **面试金句**：混合检索把向量语义检索和BM25关键词检索的结果用RRF算法融合。纯向量检索的软肋是精确词匹配——对专有名词、型号编码、法条号等，向量检索可能找不到精确匹配。纯BM25的软肋是语义理解——"竞业协议"和"非竞争条款"是同一件事，BM25无法关联。混合检索取两者之长：语义泛化+精确匹配双保险。生产环境中，混合检索通常比纯向量检索提升10-30%的命中率，是目前的主流选择。

---

**面试题4：HyDE是什么原理？它解决了什么问题？**

> 💡 **面试金句**：HyDE（假设文档嵌入）解决了"查询-文档语义空间偏差"问题。用户的问题通常是短句，而文档是长段落，两者的Embedding分布存在系统性偏差，导致检索不准。HyDE的做法是先让LLM根据用户问题生成一个假设答案，再用假设答案的Embedding去检索——假设答案在语义分布上更接近真实文档，检索精度显著提升。代价是多一次LLM调用，延迟增加。适合准确性优先、对延迟容忍度较高的场景，如离线分析、专业知识查询。

## 第五章 RAG高级技巧与失效模式 ⭐⭐⭐⭐

### 5.1 GraphRAG（微软）

---

**普通RAG的局限：多跳推理**

普通RAG擅长回答"事实查询"——找到包含答案的Chunk，直接返回。但有一类问题它处理得很差：**需要跨文档、多步推理的复杂关系查询**。

```
例子1（多跳关系）："A公司的CEO的前老板是谁？"
  → 需要：找到A公司CEO是谁（文档1）→ 找到此人的前职（文档2）
  → 普通RAG：可能只检索到其中一个文档，无法自动关联

例子2（全局摘要）："这份100页报告的核心结论是什么？"
  → 需要：理解整篇报告的全局语义，而不是某几段
  → 普通RAG：只检索出局部片段，无法整合全局视角

例子3（跨文档对比）："A公司和B公司的供应商政策有何异同？"
  → 需要：同时理解两份文档，进行结构化对比
  → 普通RAG：可能检索出两家的片段，但LLM缺乏结构化框架
```

---

**GraphRAG的核心思想**

微软于2024年发布GraphRAG，解决上述"多跳推理"和"全局理解"的局限。

```
普通RAG：
  文档 → Chunks（平铺的文本片段）→ 向量检索

GraphRAG：
  文档 → 知识图谱提取（实体+关系）→ 社区检测（聚类）→ 层级摘要
  查询时 → 图遍历 + 向量检索 → 支持关系推理
```

**知识图谱提取过程**：

```
原文："张三担任ABC公司CEO，ABC公司的母公司是XYZ集团，
      XYZ集团的董事长是李四。"

提取结果：
  实体：张三、ABC公司、XYZ集团、李四
  关系：
    张三 --[担任CEO]--> ABC公司
    ABC公司 --[隶属于]--> XYZ集团
    李四 --[担任董事长]--> XYZ集团

多跳推理："ABC公司CEO的大老板是谁？"
  → 图遍历：张三(CEO of ABC) → ABC(隶属于XYZ) → XYZ(董事长是李四)
  → 答案：李四
```

---

**GraphRAG vs 普通RAG的适用场景对比**

| 对比维度 | 普通RAG | GraphRAG |
|----------|---------|----------|
| 适合查询类型 | 事实查询、段落定位 | 关系推理、全局摘要、跨文档对比 |
| 文档结构要求 | 无特殊要求 | 文档中有丰富的实体和关系描述 |
| 构建成本 | 低（Chunking+向量化） | 高（LLM提取实体关系，成本高） |
| 查询延迟 | 低 | 高（图遍历复杂） |
| 推荐场景 | 企业FAQ、产品手册 | 组织关系分析、研究报告、复杂知识库 |

GraphRAG目前的工程挑战：知识图谱提取需要大量LLM调用，构建成本高；且对非结构化文档（散文、报告）的关系提取质量参差不齐。在大多数企业RAG场景中，普通RAG+混合检索已经足够，GraphRAG更适合有明确关系查询需求的专业场景。

---

### 5.2 父子分块（Parent-Child Chunking）

父子分块是解决"块大小两难"问题的工程最佳实践之一。

---

**问题背景：单一块大小的两难困境**

```
块太大（如1000字）：
  ✗ 一个Chunk包含多个主题，检索时匹配精度低
  ✗ 即使找到了相关Chunk，LLM还要在1000字中自己找答案
  ✓ 上下文完整，语义连贯

块太小（如100字）：
  ✓ 检索精准，能定位到具体句子
  ✗ 上下文不完整，100字的片段可能缺乏前后文，LLM理解困难
  ✗ 检索到的片段单薄，不足以支撑复杂答案
```

父子分块的洞察：**检索和上下文注入是两个不同的任务，应该用不同粒度的文本**。

---

**父子分块的工作机制**

```
建索引阶段：
  原文档
    │
    ├── 切成"父块"（Parent Chunk，512字）→ 存原始数据库
    │   [父块1: 第1-512字]
    │   [父块2: 第513-1024字]
    │
    └── 再切成"子块"（Child Chunk，128字）→ 存向量数据库
        [子块1: 第1-128字]   → 指向父块1
        [子块2: 第129-256字] → 指向父块1
        [子块3: 第257-384字] → 指向父块1
        [子块4: 第385-512字] → 指向父块1
        ...

检索阶段：
  用户问题 → 向量检索（搜子块，精准定位）
    → 找到子块3（相似度最高）
    → 通过子块3的指针，取出父块1（完整上下文）
    → 把父块1注入Prompt（而不是子块3）
```

效果：用小块（子块）的精确性找到正确位置，用大块（父块）的完整性提供上下文。两全其美。

---

### 5.3 RAG失效模式分析（4种+对应解决方案）

RAG不是银弹。理解它的失效模式，才能在系统出现问题时快速诊断和修复。

| 失效模式 | 表现症状 | 根本原因 | 解决方案 |
|----------|----------|----------|----------|
| **检索失败** | 明明知识库有答案，但LLM说"无法找到相关信息" | 分块切断了关键信息；Embedding模型对该类文本效果差；查询和文档的语义偏差大 | 改善分块策略（语义分块/父子分块）；换中文优化Embedding；用HyDE或多查询检索 |
| **检索噪音** | 检索到了不相关的Chunk，干扰LLM生成，导致答案跑偏 | Top-K设置过大引入噪音；Reranker缺失；元数据过滤缺失 | 减小Top-K；加入Reranker精排；用元数据过滤（按文档类型/时间过滤） |
| **上下文失忆** | LLM忽略了检索内容，回答了自己训练数据里的知识 | System Prompt指令不明确；检索内容与问题关联弱；Prompt中检索内容位置不佳 | 强化System Prompt指令（"必须基于以下内容回答"）；改善检索质量；把检索内容放在Prompt靠前位置 |
| **答案越界** | LLM在检索内容基础上加入了自己的推断，超出引用范围 | System Prompt约束不足；LLM的生成倾向过于"创造性" | 强化"只根据上下文回答"约束；添加"如无相关信息请明确说明"指令；降低LLM温度参数（Temperature） |

---

**RAG诊断checklist**

```
Step 1：检查检索质量
  └── 把用户问题单独做向量检索，查看Top-5结果
  └── 如果相关内容不在Top-5，问题在检索层（分块/Embedding/检索策略）

Step 2：检查Prompt质量
  └── 把检索结果+问题手动组装成Prompt，直接问LLM
  └── 如果答案还是不对，问题在Prompt设计或LLM本身

Step 3：检查生成约束
  └── 如果LLM编造内容，加强System Prompt的约束
  └── 降低Temperature（0.1-0.3）减少"创造性"
```

### 📝 第五章面试题

---

**面试题1：RAG有哪些常见的失效模式？如何诊断和解决？**

> 💡 **面试金句**：RAG有四类失效：检索失败（知识库有但找不到）、检索噪音（找到了不相关内容干扰LLM）、上下文失忆（LLM忽略检索内容）、答案越界（LLM超出检索内容编造）。诊断思路是"分层定位"：先单独测试检索质量（手动查Top-5看是否相关），确认检索正常后再测Prompt层，最后测生成层。常见修复：检索不准→改分块策略或用HyDE；噪音多→加Reranker精排；上下文失忆→强化System Prompt约束；越界→降Temperature+限制指令。

---

**面试题2：GraphRAG和普通RAG的主要区别是什么？各适合什么场景？**

> 💡 **面试金句**：核心区别在于知识表示形式。普通RAG把文档切成平铺的Chunk，通过向量相似度检索，适合事实性"找片段"查询。GraphRAG把文档提取为知识图谱（实体+关系），通过图遍历做推理，适合"A的上级的前任是谁"这类多跳关系查询，以及需要跨文档关联分析的场景。GraphRAG的代价是构建成本高（需要LLM提取实体关系），查询延迟大，对文档质量要求高。实践中，大多数企业RAG场景普通RAG+混合检索已经够用，GraphRAG在组织关系分析、复杂研究报告场景更有价值。

## 第六章 RAG vs Fine-tuning：决策框架 ⭐⭐⭐⭐⭐

> 本章是面试高频考题，也是工程选型中最容易踩坑的决策点。理解"RAG注入知识，Fine-tuning改变行为"这个核心思想，是区分有经验工程师和理论派的关键。

---

### 6.1 核心决策矩阵（8维对比）

| 决策维度 | RAG | Fine-tuning | 实际选型建议 |
|----------|-----|-------------|--------------|
| **知识更新频率** | 实时更新（修改知识库即可） | 慢（需重新训练，周级到月级） | 频繁更新选RAG，半年以上稳定知识选Fine-tuning |
| **数据量需求** | 极少（几百个文档即可运行） | 高（通常需要数百到数千条高质量标注样本） | 数据少选RAG，有充足标注数据再考虑Fine-tuning |
| **私有数据安全** | 知识不进入模型参数，数据可完全隔离 | 知识烧入参数（知识被隐式编码，但可离线推理） | 极高安全要求选RAG（数据可独立管理），离线推理选Fine-tuning |
| **风格/格式定制** | 有限（通过Prompt可部分控制） | 强（能深度改变模型的语言风格、输出格式） | 需要特定格式或专业语气选Fine-tuning |
| **推理成本** | 高（每次推理需要检索+LLM，上下文长） | 低（短Prompt推理，无检索成本） | 高并发低成本场景考虑Fine-tuning |
| **开发周期** | 短（天级，最快1天搭原型） | 长（周级，含数据准备、训练、评估） | 快速验证选RAG，产品成熟后可迁移Fine-tuning |
| **可解释性** | 高（答案有来源引用，可溯源） | 低（知识黑盒化在参数里，无法定位来源） | 需要来源引用（法律/医疗/金融合规）选RAG |
| **知识边界控制** | 精确（只用检索到的内容回答，不在库内的明确拒答） | 模糊（训练知识和预训练知识混合，边界不清晰） | 需要严格知识边界的场景选RAG |

---

**深度分析：这8个维度背后的工程洞察**

**维度1：知识更新频率——最核心的分叉点**

这是选择RAG还是Fine-tuning最重要的单一决策维度。

```
知识每天/每周变化（新闻、政策、股价）→ 只能选RAG
知识每月变化（产品手册更新）→ Fine-tuning成本过高，选RAG
知识每年稳定（古代文学风格、专业术语）→ Fine-tuning有优势
知识永久不变（特定领域的输出格式规范）→ Fine-tuning是最佳选择
```

**维度4：风格/格式定制——Fine-tuning的真正优势所在**

这个维度经常被低估。Fine-tuning改变的不是"知道什么"，而是"怎么说"。

示例对比：
- "让LLM总是用Markdown表格输出竞争分析结果"→ Fine-tuning效果远好于Prompt Engineering
- "让LLM用律师风格的语言写合同条款"→ Fine-tuning能深度内化这种风格
- "让医疗诊断助手总是先列出风险提示，再给建议"→ Fine-tuning固化这种输出结构

这些是RAG做不到的，因为RAG只负责注入知识，不改变模型的"性格"。

**维度7：可解释性——合规场景的硬约束**

金融、医疗、法律场景有一个共同的合规要求：**答案必须可溯源，必须能指出"这个结论来自哪个文件第几条"**。

RAG天然支持这个要求（每个检索Chunk都有元数据），而Fine-tuning把知识烧进参数，根本无法追溯"这句话来自哪里"。

这是为什么企业级RAG系统几乎都是RAG架构，而不是Fine-tuning：不是因为Fine-tuning效果差，而是因为合规要求下RAG是硬约束。

### 6.2 决策树

```
开始：需要给LLM增加什么能力？
                │
     ┌──────────┴──────────┐
     ▼                      ▼
 注入新知识             改变行为/风格
     │                      │
     ▼                      ▼
知识更新频繁？          Fine-tuning首选
     │是        │否          │
     ▼          ▼            │（但如果知识量也很大）
  RAG首选   数据量足够？     │
             │是    │否      ▼
             ▼      ▼   RAG + Fine-tuning结合
          Fine-tuning  RAG（数据少也能跑）
             │
             ▼
          需要来源引用？
             │是        │否
             ▼           ▼
          RAG+Fine-tuning  Fine-tuning即可
             结合
```

**简化版决策口诀**：

```
知识要实时更新？→ RAG
数据量不够？→ RAG
需要来源引用？→ RAG
要改变说话风格？→ Fine-tuning
输出格式要固化？→ Fine-tuning
部署后不想检索成本？→ Fine-tuning
```

### 6.3 两者结合的场景

RAG和Fine-tuning不是对立关系，而是互补关系。在高质量的企业AI产品中，往往是Fine-tuning负责"能力内化"，RAG负责"知识外挂"。

---

**经典结合场景1：医疗客服助手**

```
挑战：
  1. 需要用专业的医疗语气（Fine-tuning的优势）
  2. 需要实时更新的最新诊疗指南（RAG的优势）
  3. 需要精确引用来源（RAG的优势）
  4. 需要拒绝回答超出诊疗指南的问题（RAG的优势）

解决方案：
  Fine-tune → 医疗对话风格内化（专业术语、问诊格式、风险提示结构）
  RAG → 最新诊疗指南、医院科室信息、药品说明书的实时检索

Fine-tuning做什么：模型知道"我是医疗助理，我的输出格式是..."
RAG做什么：    模型知道"这个具体问题的最新答案是..."
```

**经典结合场景2：法律文书生成助手**

```
Fine-tuning：
  - 学习律所的特定写作风格（用词规范、段落结构、条款格式）
  - 学习特定法域的专业习语（如香港法律英语 vs 大陆法律中文）

RAG：
  - 最新法规、司法解释、判决案例的实时检索
  - 引用具体条款时的精确来源（第X条第X款）

结合效果：
  - 写出来的文书既有律所风格（Fine-tuning），又有准确的最新引用（RAG）
  - 两者单独使用都无法同时达到这两个目标
```

**经典结合场景3：大型软件企业内部代码助手**

```
Fine-tuning：
  - 学习企业内部代码规范（命名规则、注释风格、错误处理模式）
  - 学习内部框架的使用惯例

RAG：
  - 最新的内部API文档（每周更新）
  - 历史Bug报告和修复方案
  - 内部架构决策文档（ADR）

结合效果：
  - 生成符合公司规范的代码（Fine-tuning）
  - 使用最新的内部API（RAG）
```

---

**结合使用的工程架构**

```
用户请求
      │
      ▼
Fine-tuned LLM（基础能力+风格）
      │
      │ ← RAG检索注入（实时知识）
      ▼
最终输出

关键工程点：
  - Fine-tuning在"基础模型层"，决定模型的"性格"和"能力框架"
  - RAG在"推理层"，决定模型在具体问题上的"知识来源"
  - 两者互不干扰：Fine-tuning不影响RAG的检索机制，RAG的知识也不会因Fine-tuning而被遗忘
```

---

**反模式：不应该结合的场景**

```
❌ 用Fine-tuning来注入频繁更新的知识
   → 每次知识更新都要重训练，成本不可接受

❌ 用RAG来替代Fine-tuning做风格定制
   → Prompt Engineering可以部分实现风格控制
     但无法达到Fine-tuning深度内化的效果

❌ 没有充足标注数据就做Fine-tuning
   → 数据不足的Fine-tuning会导致过拟合和能力退化
     不如老老实实用RAG + Prompt Engineering
```

### 📝 第六章面试题（高频必考）

---

**面试题1：RAG和Fine-tuning有什么区别，如何选择？（必考）**

> 💡 **面试金句**：核心区别一句话：**RAG是给LLM配外部知识库，Fine-tuning是改变LLM本身的能力和行为**。选择框架：知识需要实时更新→RAG；需要来源引用（合规场景）→RAG；改变说话风格和输出格式→Fine-tuning；数据量不足→RAG（Fine-tuning需要高质量标注数据）。一个常见误区是用Fine-tuning注入频繁变化的知识——这是错误的，Fine-tuning不擅长知识注入，且更新成本极高。两者可以结合：Fine-tuning内化风格和能力，RAG外挂实时知识。

---

**面试题2：如何评估RAG系统的质量？有哪些指标？（RAGAS框架）**

> 💡 **面试金句**：RAG评估有专门的框架——RAGAS（RAG Assessment），核心四个指标：
>
> **忠实度（Faithfulness）**：答案是否完全基于检索内容，是否存在幻觉（LLM编造内容）。
>
> **答案相关性（Answer Relevance）**：答案是否真正回答了用户的问题，是否答非所问。
>
> **上下文精确率（Context Precision）**：检索到的Chunks中，有多少比例是真正有用的（排除噪音）。
>
> **上下文召回率（Context Recall）**：回答问题所需的关键信息，有多少比例被检索到了（有没有遗漏）。
>
> 工程上还会加：端到端延迟、检索Top-K的命中率（Hit Rate@K）、MRR（平均倒数排名）。诊断时，先看召回率定位检索问题，再看忠实度定位生成问题。

---

**面试题3：在什么情况下RAG和Fine-tuning可以结合使用？**

> 💡 **面试金句**：当一个应用同时有两类需求时，结合使用最合适：一类是"行为/风格"需求（Fine-tuning擅长），另一类是"实时/私有知识"需求（RAG擅长）。典型场景：医疗客服需要专业医疗语气（Fine-tuning）+最新诊疗指南（RAG）；法律助手需要律所写作风格（Fine-tuning）+最新司法解释（RAG）；企业代码助手需要代码规范（Fine-tuning）+最新内部API文档（RAG）。结合的工程思路：Fine-tuning在基础模型层定义"性格"，RAG在推理层注入"知识"，两者互不干扰。

---

**面试题4：RAG的检索精度如何提升？（综合题，考察系统思维）**

> 💡 **面试金句**：这道题考察的是系统化思维，不是某个单点技巧。提升检索精度要从三个层次着手：
>
> **层次一：分块优化（影响最大）**
> - 用递归字符分块替代固定大小分块，保证语义完整
> - 设置合理的overlap（10-20%），防止边界信息丢失
> - 高质量场景改用语义分块或父子分块
>
> **层次二：检索策略优化**
> - 纯向量检索升级为混合检索（向量+BM25），提升关键词命中率
> - 引入HyDE，减少查询和文档的语义分布偏差
> - 用多查询检索提升召回率（多角度覆盖）
>
> **层次三：重排序优化**
> - 在检索后加Reranker（BGE-Reranker或Cohere Rerank）
> - 粗排海选 + 精排精选，兼顾速度和质量
>
> 诊断优先级：先解决分块问题（往往是最大的瓶颈），再解决检索策略，最后加Reranker。
